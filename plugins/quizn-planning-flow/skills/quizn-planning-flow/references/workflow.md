# Workflow · Versioning · Project Folder

워크플로우, 버전 규칙, 프로젝트 폴더 요건을 정의한다. (절대 규칙 6·7 담당)

---

## 1. 프로젝트 단위 진행 (Project-scoped) — 절대 규칙 7

모든 작업은 **반드시 프로젝트 폴더 안에서** 진행한다. 기획 요청을 받으면 가장 먼저 프로젝트
폴더가 있는지 확인하고, 없으면 아래 요건으로 생성한 뒤에만 다음 단계로 넘어간다.

### 1.1 프로젝트 폴더 생성 요건 (필수)

신규 프로젝트는 아래 구조를 **반드시** 갖춘다. 폴더명은 `QNP_{YYYYMMDD}_{프로젝트슬러그}` 형식.
(QNP = QuizN Planning)

```
QNP_20260617_show-live-onboarding/
├── PROJECT.md              # 프로젝트 메타 (필수): 제목, 서비스, 오너, 현재 버전, 상태
├── CHANGELOG.md            # 버전 변경 이력 (필수)
├── 01_evidence/            # 증거·근거 자료 (근거표, 리서치, 인터뷰, 데이터)
│   └── evidence-table.md
├── 02_requirements/        # 요구사항·정책 정의 (기획서/PRD)
│   └── plan_v0.1.0.md
├── 03_flowcharts/          # 흐름도 SVG 및 HTML 산출물
│   ├── src/                # *.html (SVG 포함), *.svg 원본
│   └── exports/            # png-fhd/, png-4k/, pdf/ 내보내기 결과
├── 04_verification/        # 논리 검증 리포트 (오류·대안)
│   └── verify_v0.1.0.md
├── 05_decisions/           # 질의-응답·수용 결정 기록 (Q&A, 의사결정)
└── assets/                 # planning-flow.css + colors_and_type.css + fonts/
```

> 폴더 생성 후 사용자에게 경로를 안내한다. 폴더 생성 없이 흐름도/기획서를 단독으로 만들지 않는다.
> 단, 사용자가 "단발성 산출물"을 명시하면 최소 `PROJECT.md` + `CHANGELOG.md`만 둔 경량 폴더로 진행.

### 1.2 PROJECT.md 필수 항목

| 항목 | 예시 |
|---|---|
| 프로젝트명 | SHOW 라이브 온보딩 개선 |
| 대상 서비스 | SHOW (data-theme=show) |
| 오너/요청자 | 홍길동 |
| 현재 버전 | v0.3.1 |
| 상태 | Draft / In-Review / Accepted / Released |
| 생성일 / 최종수정 | 2026-06-17 / 2026-06-17 |
| 기본 모드 | light + dark 동시 지원 |

## 2. 버전 표기 규칙 (Versioning) — 절대 규칙 6

**모든 진행/확정 산출물에 버전을 표기**하고, 수정·변경 시 **반드시 버전을 올린다(bump)**.

### 2.1 버전 체계 — SemVer 기반 `vMAJOR.MINOR.PATCH`

| 자리 | 올리는 경우 | 예 |
|---|---|---|
| **MAJOR** | 기획 방향/범위 변경, 흐름도 구조 재설계 등 호환 불가 변경 | v1.0.0 → v2.0.0 |
| **MINOR** | 단계/분기/요구사항 추가 등 기능적 확장(하위호환) | v1.2.0 → v1.3.0 |
| **PATCH** | 문구 수정, 라벨/오타, 색·여백 등 사소한 변경 | v1.2.0 → v1.2.1 |

- 최초 작성은 **v0.1.0 (Draft)**. 요청자 수용 전까지는 0.x.y 유지.
- 요청자가 정식 수용하면 **v1.0.0 (Accepted)** 로 승격.
- 사전 릴리스가 필요하면 `-rc.1`, `-draft` 등 접미사 허용: `v1.0.0-rc.1`.

### 2.2 버전 스탬프 위치 (어디에 표기하나)

버전은 다음 모든 위치에 **동기화**해 표기한다.

1. 산출 파일명: `plan_v0.3.1.md`, `flow_login_v0.3.1.html`
2. 문서 상단 버전 헤더(아래 양식)
3. 흐름도 SVG 우하단 버전 배지(`#version-badge`, 템플릿에 내장)
4. `PROJECT.md`의 현재 버전
5. `CHANGELOG.md`의 새 항목

### 2.3 문서 버전 헤더 양식 (모든 .md 산출물 상단)

```
> **버전 / Version:** v0.3.1
> **상태 / Status:** In-Review
> **최종 수정 / Updated:** 2026-06-17
> **변경 요약 / Change:** 결제 실패 분기 추가, 에러 토스트 문구 수정
> **이전 버전 / Prev:** v0.3.0
```

### 2.4 CHANGELOG.md 양식

```
# CHANGELOG — {프로젝트명}

## v0.3.1 — 2026-06-17  (In-Review)
### Changed
- 결제 실패 시 재시도 분기 추가 (verify_v0.3.1 #E2 대안 채택)
### Fixed
- 에러 토스트 문구 오타 수정
### Verification
- 논리 검증: 오류 2건 → 대안 채택 1, 보류 1 (04_verification/verify_v0.3.1.md)
### Decision
- 요청자 질의 #Q4 수용 (05_decisions/2026-06-17.md)

## v0.3.0 — 2026-06-16  (Draft)
...
```

> **변경이 발생했는데 버전을 올리지 않는 것은 규칙 위반**이다. 사소한 수정도 PATCH를 올린다.

## 3. 단계별 게이트 (Stage Gates)

| 게이트 | 통과 조건 |
|---|---|
| G1 (증거) | 핵심 요구마다 근거 또는 `[가정]` 표시가 있다 |
| G2 (흐름도) | SVG로 작성됐고, 다크/라이트·FHD~4K 요건을 충족한다 |
| **G3 (검증)** | 논리 검증 리포트(오류+대안)가 작성됐다 — **필수, 생략 불가** |
| **G4 (질의·수용)** | 검증 결과를 요청자에게 질의했고, 답변을 반영해 수용 결정을 기록했다 |
| G5 (버전) | 버전이 모든 위치에 동기화 표기되고 CHANGELOG가 갱신됐다 |

G3·G4를 통과하지 않은 흐름도는 "확정"이라 부르지 않는다. "초안(Draft)"으로만 표기한다.

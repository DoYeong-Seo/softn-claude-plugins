# Logic Verification — 논리 검증 (절대 규칙 2·4·5)

흐름도를 그린 **직후 반드시** 논리 검증을 수행한다. 검증 결과(오류 리포트 + 대안)를
요청자에게 **질의**하고, 답변을 반영한 뒤에만 수용(확정)한다. 검증·질의 없이 확정 금지.

순서: **그린다 → 검증한다 → (오류 리포트 + 대안) → 요청자에게 질의 → 수용**

---

## 1. 논리 검증 체크리스트 (Logic Checklist)

흐름도/프로세스의 논리 구조를 아래 항목으로 점검한다.

### 1.1 구조 (Structure)
- [ ] **시작/종료**: 단일 시작점이 있고, 모든 경로가 명시적 종료로 수렴하는가?
- [ ] **고립 노드**: 들어오거나 나가는 화살표가 없는 노드가 없는가?
- [ ] **연결성**: 시작점에서 모든 노드에 도달 가능한가? (도달 불가 노드 = orphan)

### 1.2 분기 (Decisions)
- [ ] **분기 완전성(MECE)**: 결정 다이아몬드의 분기들이 상호배타·전체포괄인가? (Yes/No 누락 없는가)
- [ ] **분기 라벨**: 모든 분기 화살표에 조건 라벨이 있는가?
- [ ] **중복 조건**: 같은 입력에 둘 이상 분기가 동시에 참이 되는 모순이 없는가?

### 1.3 흐름 (Flow)
- [ ] **무한 루프**: 빠져나갈 종료 조건 없는 순환이 없는가?
- [ ] **데드엔드**: 종료가 아닌데 더 진행할 화살표가 없는 막다른 노드가 없는가?
- [ ] **경합/순서**: 선행조건 없이 실행되는 단계가 없는가? (순서 의존성 위반)

### 1.4 예외·경계 (Edge & Error)
- [ ] **에러 경로**: 실패/타임아웃/취소/권한없음 등 예외 경로가 정의돼 있는가?
- [ ] **경계값**: 빈 입력·최대치·중복 제출 등 경계 케이스가 처리되는가?
- [ ] **롤백/보상**: 실패 시 상태 복구(롤백) 또는 보상 트랜잭션이 있는가?

### 1.5 정합성 (Consistency)
- [ ] **증거 정합**: 각 결정이 근거표(EV-ID)와 모순되지 않는가?
- [ ] **정책 정합**: 보안/약관/결제/개인정보 등 정책 요건과 충돌하지 않는가?
- [ ] **용어 일관**: 같은 개념을 한 용어로 부르는가? (라벨 표준화)

## 2. 논리 오류 분류 (Error Taxonomy) — 리포트 시 사용

| 코드 | 유형 | 설명 |
|---|---|---|
| L1 | Orphan/Unreachable | 도달 불가·고립 노드 |
| L2 | Dead-end | 종료 아닌 막다른 노드 |
| L3 | Non-MECE branch | 분기 누락/중복(비배타) |
| L4 | Missing label | 라벨 없는 분기 |
| L5 | Infinite loop | 종료 조건 없는 순환 |
| L6 | Missing error path | 예외/실패 경로 부재 |
| L7 | Order violation | 선행조건 위반·경합 |
| L8 | Edge case gap | 경계값 미처리 |
| L9 | Evidence/Policy conflict | 근거·정책과 모순 |
| L10 | Terminology drift | 용어 불일치 |

심각도: **Critical**(흐름이 깨짐) / **Major**(주요 누락) / **Minor**(사소·표기).

---

## 3. 검증 리포트 양식 (필수) — `04_verification/verify_v{버전}.md`

논리 오류는 **0건이어도 반드시 리포트**한다(절대 규칙 4). 발견된 오류마다 **대안을 1개 이상**
함께 제시한다(절대 규칙 5).

```
> **버전 / Version:** v0.3.1   **검증일:** 2026-06-17   **대상:** flow_login_v0.3.1

## 검증 요약 (Summary)
- 점검 항목: 15 / 통과: 13 / 오류: 2 (Critical 1, Major 1, Minor 0)
- 결론: 요청자 질의 필요 (아래 #E1, #E2)

## 발견 오류 및 대안 (Findings & Alternatives)

### #E1  [Critical] L6 — 결제 실패 경로 없음
- 위치: "결제 요청" → "완료" 사이
- 문제: 결제 실패/타임아웃 시 흐름이 정의되지 않아 사용자가 멈춤.
- 영향: 결제 이탈·CS 문의 증가 (근거: EV-03)
- **대안 A (권장):** 실패 시 "재시도(최대 3회) → 실패 시 대체 결제수단 안내" 분기 추가.
- **대안 B:** 실패 시 즉시 장바구니 보존 후 종료, 이메일 리마인드.
- 트레이드오프: A는 전환율↑·구현비↑ / B는 단순·전환율↓.

### #E2  [Major] L3 — 닉네임 검증 분기 비-MECE
- 문제: "중복" 분기는 있으나 "금칙어/길이초과" 케이스가 누락.
- **대안 A (권장):** 검증을 [중복/금칙어/길이/허용] 4분기로 확장.
- **대안 B:** 단일 "검증 실패" 분기로 묶고 사유는 토스트로 표기.

## 통과 항목 (Passed) — 13건
- L1 고립 노드 없음 / L2 데드엔드 없음 / ... (요약)

## 오류 없음일 때
- "전 항목 통과. 논리 오류 0건. (검증 완료)" 를 명시한다.
```

## 4. 요청자 질의 양식 (Question to Requester) — `05_decisions/`

검증 후 요청자에게 **선택형 질의**를 한다. 각 오류의 대안을 보기로 제시하고 결정을 받는다.

```
## 질의 #Q4 — 결제 실패 처리 (verify #E1 관련)
선택해 주세요:
( ) 대안 A: 재시도 3회 후 대체 결제수단  [권장]
( ) 대안 B: 장바구니 보존 + 이메일 리마인드
( ) 기타: ____________________

## 수용 결과 (Decision)
- 채택: 대안 A
- 사유: 전환율 우선 (근거: EV-01)
- 반영 버전: v0.3.1 → 흐름도/문서 bump, CHANGELOG 기록
```

질의는 가능하면 AskUserQuestion 도구로 선택지를 제시한다. 답변을 받으면 흐름도를 수정하고
**버전을 bump**한 뒤 CHANGELOG·decisions에 기록한다.

---

## 5. 진행 규칙 요약

1. 흐름도 작성 → **즉시 검증** (생략 불가)
2. 오류 리포트 작성 → **각 오류마다 대안 ≥1** (생략 불가)
3. 요청자 **질의** → 답변 수집
4. 답변 반영 → 흐름도 수정 → **버전 bump** → CHANGELOG·decisions 기록 → 수용 확정

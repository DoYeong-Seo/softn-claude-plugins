# Evidence-based Planning — 증거 기반 기획 (절대 규칙 1)

기획의 모든 주장·결정에는 **근거**가 있어야 한다. 근거가 없는 것은 `[가정]`으로 명시하고
검증 대상으로 분류한다. "그럴 것 같다"는 표현만으로 흐름도/요구사항을 확정하지 않는다.

---

## 1. 근거 등급 (Evidence Grades)

| 등급 | 의미 | 예시 |
|---|---|---|
| **E-A** | 1차 정량 데이터 | 제품 분석(GA/Amplitude), 로그, A/B 테스트 결과 |
| **E-B** | 1차 정성 데이터 | 사용자 인터뷰, 설문, CS/문의 로그, 세션 리플레이 |
| **E-C** | 내부 합의·정책 | 확정된 정책 문서, 의사결정 기록, 법무/보안 요건 |
| **E-D** | 2차 자료 | 경쟁사 분석, 업계 벤치마크, 외부 리서치 |
| **[가정]** | 근거 미확보 | 아직 검증되지 않은 추정 — 반드시 표시·검증 큐로 |

원칙: **핵심 결정일수록 높은 등급(E-A/E-B/E-C)** 의 근거를 요구한다. 낮은 등급·가정만으로
핵심 흐름을 확정하지 않는다.

---

## 2. 근거표 (Evidence Table) — `01_evidence/evidence-table.md`

요구사항·기획 결정마다 한 행으로 근거를 연결한다.

| ID | 요구/주장 (Claim) | 근거 등급 | 출처 (Source/Link) | 신뢰도 | 비고 |
|---|---|---|---|---|---|
| EV-01 | 신규 사용자 40%가 첫 라이브 진입 전 이탈 | E-A | Amplitude 퍼널 2026-05 | 높음 | 표본 12,430 |
| EV-02 | 닉네임 입력 단계가 가장 큰 이탈 지점 | E-A | 동일 퍼널 step3 | 높음 | |
| EV-03 | "방 코드가 헷갈린다"는 불만 다수 | E-B | CS 문의 38건/4주 | 중간 | |
| EV-04 | 자동 닉네임 제안 시 이탈 감소 | [가정] | — | — | A/B 검증 필요 |

> 흐름도/요구사항 문서에서 결정을 서술할 때는 `(근거: EV-01, EV-02)`처럼 ID를 인용한다.

## 3. 가정 관리 (Assumption Log)

`[가정]`으로 표시된 항목은 별도 큐로 관리하고, 검증 방법·기한·담당을 적는다.

| 가정 ID | 내용 | 검증 방법 | 기한 | 상태 |
|---|---|---|---|---|
| AS-01 | 자동 닉네임이 이탈을 낮춘다 | A/B 테스트(2주) | 2026-07-01 | 진행중 |

가정이 검증되면 근거표로 승격(등급 부여)하고, 관련 흐름도/문서 **버전을 bump** 한다.

---

## 4. 증거 기반 작성 체크리스트

흐름도·기획서를 확정하기 전 아래를 확인한다.

- [ ] 모든 핵심 요구사항이 근거표 ID와 연결돼 있다.
- [ ] 근거 없는 주장은 모두 `[가정]`으로 표시했다.
- [ ] 핵심 결정은 E-A/E-B/E-C 근거를 갖는다(또는 가정으로 명시 + 검증 큐 등록).
- [ ] 출처에 접근 경로(링크/문서명/날짜)가 적혀 있다.
- [ ] 상충하는 근거가 있으면 함께 기록하고 판단 근거를 남겼다.

> 데이터 출처가 사내 시스템(Amplitude, GA, Notion, Slack, CS 도구 등)일 경우, 가능하면
> 해당 커넥터로 직접 확인해 출처 링크를 남긴다. 외부 자료는 출처 URL과 확인 날짜를 적는다.

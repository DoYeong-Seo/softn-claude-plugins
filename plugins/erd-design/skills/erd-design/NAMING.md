# ERD 명명 워크플로우 — 용어사전·도메인 통합

테이블·컬럼을 **생성 또는 수정**할 때 반드시 이 절차를 따른다. [SKILL.md](SKILL.md)는 본 문서 절차로 위임한다. 모델이 임의로 `physicalName`을 만들거나 도메인을 골라서는 안 된다.

## 핵심 원칙 — 절대 위반 금지

1. **모든 `physicalName`은 용어사전(Glossary) 매핑으로 조립**한다. 단일 토큰이든 복합어든, 각 토큰은 사전 등록된 `physicalName`을 사용한다. 임의 축약·번역 금지.
2. **사전에 없는 용어는 등록 절차를 먼저 거친다** — `AskUserQuestion`(또는 동등한 명시적 확인)으로 **사용자 동의를 받기 전에는 절대 등록하지 않는다**. 거부·보류된 토큰이 있으면 해당 테이블/컬럼 작업 자체를 중단한다.
3. **컬럼은 반드시 `domainId`와 함께 등록**한다 — 논리명 기반으로 후보를 추천하되, **사용자가 직접 선택**해야 한다. 모델이 임의로 결정 금지. "도메인 없음" 도 명시적 선택지로 제공.

## 사전 준비 (작업 단위마다 1회)

테이블·컬럼 작업 시작 전 다음 3건을 한 번에 조회하여 메모리에 보관 후 재사용한다(반복 조회 금지):

| 조회 | 엔드포인트 | 용도 |
|------|----------|------|
| 프로젝트 상세 | `GET /api/v1/erd/project/{projectId}` | `dictionaryId` 획득 |
| 용어 목록 | `GET /api/v1/erd/glossary/{dictionaryId}/list` | 논리명↔물리명 매핑 |
| 도메인 목록 | `GET /api/v1/erd/project/{projectId}/domain/list` | 컬럼 타입 후보 |

프로젝트에 `dictionaryId`가 없으면 사용자에게 사전을 먼저 생성하도록 안내(`POST /api/v1/erd/glossary/dictionary`) — 사전 없이는 워크플로우 진행 불가.

## 절차

### STEP 1. 논리명 분해
- 입력된 한국어 논리명을 의미 단위 토큰으로 쪼갠다. 예: `공지사항 제목` → `공지사항`, `제목`.
- `[업무영역]` / 끝의 `]` 약어 마커는 [CONVENTIONS.md](CONVENTIONS.md) 규칙을 따른다 (예: `정보]` = `INFO`).

### STEP 2. 사전 조회 → 누락 보고 → 등록 동의
- 각 토큰을 glossary `logicalName`과 매칭.
- **하나라도 누락**되면 작업을 일시 중단하고, 누락 토큰을 모아 사용자에게 보고:
  ```
  사전에 없는 용어가 있습니다. 등록하시겠습니까?
    1) 공지사항    → 제안 physicalName: NOTICE    설명: ...
    2) 제목       → 제안 physicalName: TITLE     설명: ...
  각 항목을 등록/수정/거부 중 선택해주세요. 동의 없이는 진행하지 않습니다.
  ```
- **반드시 `AskUserQuestion` 도구로 각 누락 토큰의 등록 여부와 최종 `physicalName`/`description`을 확정**한다. 모델 제안은 초안일 뿐이며, 사용자가 수정·승인할 때까지 절대 `POST`하지 않는다.
- 동의받은 항목만 `POST /api/v1/erd/glossary/{dictionaryId}` 로 하나씩 등록.
- 거부·보류 항목이 남으면 해당 테이블/컬럼 작업 자체를 보류하고 사용자에게 보고. "사전 우회"는 옵션이 아니다.

### STEP 3. 물리명 조립
- 사전 `physicalName`을 `_` 로 연결. 예: `공지사항(NOTICE)` + `제목(TITLE)` → `NOTICE_TITLE`.
- 약어·업무영역 마커는 사전 매핑값을 그대로 사용. 임의 축약 금지.
- 조립 결과가 모호하거나 부자연스러우면 STEP 2로 돌아가 사전 항목을 재검토.

### STEP 4. 도메인 매칭 (컬럼 한정)
- 도메인 목록에서 논리명·물리명·용도에 부합하는 후보 1~3개를 추출하여 사용자에게 제시.
  - 추천 휴리스틱(참고용, 최종 결정은 사용자): `*_ID`/`*키` → ID 도메인, `*명`/`*제목` → NAME 도메인, `*일시`/`*날짜` → DATETIME 도메인, `*여부`/`*FLAG` → FLAG 도메인 등.
- **`AskUserQuestion` 으로 사용자가 직접 선택**한다. "도메인 없음" 도 선택지에 반드시 포함.
- 적합한 도메인이 없다고 사용자가 판단하면 새 도메인 생성 흐름(`POST /api/v1/erd/project/{projectId}/domain`) 을 제안하고, 신규 도메인의 스펙(`domainName`/`dataType`/`length`/`precision`/`scale`)도 사용자에게 확인.
- 선택된 `domainId`는 컬럼 생성·수정 body에 **필수로 포함**한다 (없음 선택 시에만 생략).

### STEP 5. 최종 요약 → 호출
- 호출 직전 사용자에게 최종 요약(테이블 논리명/물리명, 컬럼별 논리명/물리명/`domainId`)을 보여주고 동의 확인. 동의 후 `POST/PUT` 실행.

## 호출 전 체크리스트

- [ ] 프로젝트의 `dictionaryId`를 확인했는가? (없으면 사용자에게 사전 생성 안내)
- [ ] 모든 논리명 토큰이 glossary에 등록되어 있는가? (누락 시 사용자 승인 후 등록 완료)
- [ ] `physicalName`은 100% 사전 매핑에서 조립되었는가? (임의 축약·번역 0건)
- [ ] 모든 컬럼이 사용자가 명시적으로 선택한 `domainId`(혹은 "없음")를 가지고 있는가?
- [ ] 사용자에게 최종 변경 요약을 보여주고 동의를 받았는가?

## 워크플로우 비대상

- 삭제(DELETE) — 명명 검증 불필요.

# 세션 간 협업 프로토콜 (peer-bus)

> 이 블록을 백엔드/프론트 각 프로젝트의 `CLAUDE.md`에 붙여넣는다.
> 핵심 원칙: **계약 본문은 BlogN, 버스는 트리거(포인터)만.** 본문을 버스에 싣지 않는다.

## 0. 정체성 · 주소
- 내 식별자는 채널 노드의 `NAME` 값이다: `backend` / `front:blogn` / `front:erd` …
- BlogN 분류 규칙: 계약·요청 = `agent-contracts`, 응답 = `agent-responses`.
- **백엔드 = 코디네이터**: API 계약·스키마 결정의 단일 진실(SoT)을 소유한다.

## 1. 세션 시작 시 (동기화)
- 등록 직후 **현재 유효 계약 포인터**(refPostId 목록)를 받는다.
  - push 모드: `<channel ... replay="1" refPostId=...>` 이벤트로 자동 도착.
  - poll 모드: 첫 `read_messages` 호출 결과에 `replay` 표시로 함께 온다.
- 각 `refPostId`를 BlogN `get_post` 로 읽어 현재 계약 상태를 동기화한다.
- (교체된 프론트라면 이 단계만으로 직전 계약을 즉시 따라잡는다 — board replay 불필요.)

## 2. 메시지 수신 시 (`{from, type, refPostId}`)
- **type=req** (요청): `refPostId`를 BlogN에서 읽는다 → **계획 또는 거부**를 결정해
  `agent-responses` 분류로 응답 포스트 작성(`create_post` + `update_post_contents`)
  → `send_to(to=from, type="resp", refPostId=<새 응답글 ID>)`.
- **type=resp** (응답): `refPostId`를 읽고 반영 **계획**을 정리해 사람에게 보고한다.
  실제 코드 변경·머지는 **사람 승인 후**에만(§4).
- **type=notice** (알림): 참고만. 필요 시 계약 동기화.

## 3. 메시지 송신 시
- 계약/요청을 만들면 **먼저 BlogN에 포스트로 쓴다**(긴 본문·스펙·수용기준은 전부 여기).
- 그 다음 `broadcast` 또는 `send_to` 로 `{type, refPostId, body?}` **트리거만** 보낸다.
- 그 계약이 "이후로도 유효한 현재 계약"이면 `pin: true` 로 보내 허브 핀 인덱스에 등록한다.
  계약이 폐기되면 같은 `refPostId` 로 `pin: false`.

## 4. 사람 승인 게이트 (필수)
- 자동으로 진행해도 되는 범위: **포스트 읽기 + 계획/거부 문서 작성 + 트리거 송신**까지.
- **코드 수정·파일 변경·커밋·머지·배포는 사람 승인 후**에만 실행한다.
- 위험한 Bash가 필요하면, 권한 릴레이를 켠 경우 다른 세션(예: backend)에서 승인받는다.

## 5. 루프 규율
- 위 1~4를 자율적으로 이어가되, 사용자에게 "다음에 뭐 할까요"를 매번 되묻지 않는다.
- 단, §4 게이트에 걸리는 작업은 반드시 멈추고 승인을 요청한다.
- 동료가 누구인지 모르면 `peers` 도구로 현재 접속자를 확인한 뒤 보낸다.

## 6. 폴링 모드 수신 루프 (PEER_BUS_MODE=poll, 기본)
- push(Channels) 프리뷰·개발 플래그 없이 운영하는 기본 모드다.
- **작업의 자연스러운 멈춤마다 `read_messages` 를 호출**해 새 트리거를 가져온다
  (요청 처리를 끝냈을 때, 사용자 보고 직전, 유휴 진입 시).
- `read_messages` 가 트리거를 반환하면 §2 로 처리하고, 처리 후 다시 `read_messages` 로 이어
  받는다. `(새 메시지 없음)` 이면 현재 작업/유휴 상태를 유지한다.
- 도구 호출이 반환되면 턴이 끝나므로, 연속 자율 루프가 필요하면 사용자가 한 번 밀어주거나
  push 모드로 승격한다. 놓친 메시지는 본문이 BlogN에 남아 재조회로 복구된다.
